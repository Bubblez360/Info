### Deployment sequence
```
1. git push to main
2. Vercel webhook fires → vite build (content.ts + snapshot.json bundled)
3. Build passes schema validation (Vitest gate) — fails closed if content invalid
4. Static assets deployed to Vercel edge
5. Post-deploy: manual smoke check (home loads, theme toggle, contact form visible/hidden correctly)

Separately, weekly (Sunday 00:00 UTC):
1. GitHub Action runs, fetches GitHub API
2. Commits public/snapshot.json if changed
3. Commit triggers step 1-5 above automatically
```

### Rollback
```
Bad deploy on Vercel → Vercel dashboard → "Promote to Production" on last-known-good
deployment (instant, no rebuild). Or: git revert <commit> → push → auto-redeploy.
Reversibility: 5/5 — fully static site, no DB, no migrations, no stateful backend.
```

## Worktree Parallelization Strategy

| Step | Modules touched | Depends on |
|---|---|---|
| Scaffold + design tokens | root config, src/styles/, PRODUCT.md/DESIGN.md | — |
| Content layer + schema | src/data/ | Scaffold |
| Core sections (Hero/About/Skills/Certs/Projects) | src/components/sections/ | Content layer |
| Resume + Contact | src/components/resume/, src/components/contact/ | Content layer |
| GitHub viz + snapshot Action | src/components/github/, scripts/, .github/workflows/ | Content layer |
| Command palette (cmdk) | src/components/command-palette/ | Core sections (needs nav targets) |
| Gen-art canvas | src/components/hero/ | Scaffold only |
| SEO/app-shell (OG, 404, error boundary, favicon) | public/, index.html | Scaffold only |
| Tests | src/**/*.test.ts, e2e/ | All above (tests follow implementation) |

**Lane A (parallel, independent):** Gen-art canvas + SEO/app-shell — touch disjoint modules (`src/components/hero/` vs `public/`+`index.html`), no shared dependency.
**Lane B (sequential):** Scaffold → Content layer → {Core sections, Resume+Contact, GitHub viz} → Command palette.
**Execution order:** Launch Lane A in a parallel worktree once Scaffold lands. Lane B proceeds sequentially in the main worktree. Merge Lane A back after Core sections land (command palette needs nav targets from Core sections regardless). Tests written alongside each step, not deferred.
**Conflict flag:** none — Lane A and Lane B touch disjoint module directories.

Given this is a single-developer AI-assisted build (not a human team to split across worktrees), sequential execution in one session is the practical default — the lanes above describe *logical* independence for reference, not a recommendation to actually spawn parallel worktrees for a project this size.
